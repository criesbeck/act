A cause-effect diagram is used in retrospectives to analyze the causal forces leading to recurring issues in the development process.
The purpose of the diagram is to allow teams to start with some repeating event that bothers them, but the identify
  • the metrics that matter that the event affects
  • the causal factors that cause the event to repeat
  • the best points at which to introduce a process change
It's important to move the conversation to the metrics that matter. Many retrospectives start with obvious issues, such as poor attendance, rambling meetings, or ugly code. But those issues must be connected to one or more metrics that matter, in order to determine how to prioritize when to address the issue, with what level of effort, to what level of improvement.
Henrik Kniberg's article on cause effect diagrams (PDF) for root cause analysis is an excellent tutorial with examples on this tool.
